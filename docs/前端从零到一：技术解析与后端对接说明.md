# 前端从零到一：技术解析与后端对接说明

> 面向前端零基础读者，讲清楚"我们加了一个前端"究竟加了什么、每个技术是干什么的、代码是怎么工作的。

---

## 一、先建立一个基础认知：前端是什么

用最直白的话说：

- **后端**：藏在服务器里的大脑，处理数据、调用 AI、操作数据库，但它只会"说话"（返回 JSON 数据），不会"画画"（渲染界面）。
- **前端**：运行在用户浏览器里的界面程序，负责把后端返回的 JSON 数据变成用户能看、能点、能输入的页面。

两者的通信方式：前端通过 HTTP 请求向后端"问数据"，后端返回 JSON，前端再把 JSON 渲染成 HTML。

```
用户浏览器                  服务器
┌──────────────┐   HTTP    ┌──────────────┐
│  React 前端   │ ────────> │ FastAPI 后端  │
│  （渲染界面） │ <──JSON── │（处理业务逻辑）│
└──────────────┘           └──────────────┘
```

## 二、我们用了哪些前端技术

### 2.1 技术栈全览

| 技术 | 类比理解 | 作用 |
|------|---------|------|
| **Node.js** | 前端的"Python 解释器" | 运行 JavaScript 工具链的环境（不是后端！） |
| **npm** | 前端的"pip" | 管理前端依赖包 |
| **Vite** | 前端的"uvicorn" | 开发服务器 + 打包构建工具 |
| **TypeScript** | 带类型的 JavaScript | 给 JS 加上类型检查，减少 bug |
| **React** | 界面组件库（核心框架） | 用"组件"的方式构建界面 |
| **Ant Design** | UI 组件库 | 提供现成的按钮、表格、弹窗等组件 |
| **React Router** | URL 路由管理 | 决定不同 URL 显示哪个页面 |
| **Zustand** | 前端的"全局变量管理器" | 跨组件共享状态（如登录 Token） |
| **Axios** | 前端的"httpx / requests" | 封装 HTTP 请求 |
| **react-markdown** | Markdown 渲染库 | 把 AI 回复的 Markdown 文本渲染成富文本 |

### 2.2 重点技术详解

#### React：一切皆组件

React 的核心思想是把界面拆成独立的"组件（Component）"，每个组件就是一个函数，接受参数、返回 HTML。

```tsx
// 最简单的 React 组件
function Hello({ name }: { name: string }) {
  return <div>你好，{name}</div>
}
```

组件内部有两个核心概念：
- **State（状态）**：组件自己的变量，变了就重新渲染界面。用 `useState` 创建。
- **Effect（副作用）**：组件渲染完后执行的操作（如请求接口）。用 `useEffect` 创建。

#### Vite：开发服务器 + 打包工具

Vite 做两件事：
1. **开发时**：启动一个本地服务器（`http://localhost:3000`），支持热更新（改代码浏览器自动刷新）。
2. **生产时**：把所有代码打包成几个静态文件（`dist/` 目录），可以放到任何 Web 服务器上。

#### TypeScript：JavaScript + 类型

TypeScript 在 JavaScript 基础上加了类型声明，类比 Python 的 type hint：

```typescript
// JavaScript（无类型）
function greet(user) { return "Hello " + user.name }

// TypeScript（有类型，接口清晰）
interface User { name: string; email: string }
function greet(user: User): string { return "Hello " + user.name }
```

## 四、每个模块做了什么

### 4.1 程序入口 `src/main.tsx`

这是 React 应用的"电源开关"，做三件事：

1. 把 React 应用挂载到 `index.html` 里的 `<div id="root">` 节点上
2. 注入 Ant Design 的全局配置（主题色 `#4f46e5`、中文语言包）
3. 注入 React Router 的路由系统

```tsx
ReactDOM.createRoot(document.getElementById('root')!).render(
  <ConfigProvider locale={zhCN} theme={{ token: { colorPrimary: '#4f46e5' } }}>
    <App>
      <RouterProvider router={router} />
    </App>
  </ConfigProvider>
)
```

### 4.2 路由 `src/router/index.tsx`

路由就是"URL → 页面"的映射表。本项目有：

| URL | 对应页面 | 是否需要登录 |
|-----|---------|------------|
| `/login` | 登录/注册页 | 否 |
| `/agents` | 智能体列表 | 是 |
| `/chat/:agentId` | 对话页 | 是 |
| `/`（根路径） | 自动跳转 `/agents` | 是 |

其中 `RequireAuth` 是一个"守卫组件"：访问受保护路由时，如果 store 里没有 Token，直接重定向到 `/login`。

```tsx
function RequireAuth() {
  const token = useAuthStore((s) => s.token)
  if (!token) return <Navigate to="/login" replace />
  return <Outlet />   // 有 token，正常渲染子页面
}
```

### 4.3 全局状态 `src/store/auth.ts`

使用 **Zustand** 存储登录 Token 和用户信息，并通过 `persist` 中间件把数据持久化到浏览器的 `localStorage`——这样刷新页面不会掉登录状态。

```typescript
export const useAuthStore = create<AuthState>()(
  persist(
    (set) => ({
      token: null,
      user: null,
      setAuth: (token, user) => set({ token, user }),   // 登录时调用
      clearAuth: () => set({ token: null, user: null }), // 登出时调用
    }),
    { name: 'abp-auth' },  // localStorage 存储的 key
  )
)
```

任何组件都可以通过 `useAuthStore((s) => s.token)` 读取 Token，不需要通过 props 层层传递。

### 4.4 HTTP 客户端 `src/api/client.ts`

对 Axios 进行了两层封装：

**请求拦截器**（每个请求发出前自动执行）：
```typescript
client.interceptors.request.use((config) => {
  const token = useAuthStore.getState().token
  if (token) config.headers.Authorization = `Bearer ${token}`  // 自动加上 JWT
  return config
})
```

**响应拦截器**（收到响应后自动执行）：
```typescript
client.interceptors.response.use(
  (res) => res,
  (err) => {
    if (err.response?.status === 401) {
      useAuthStore.getState().clearAuth()   // Token 失效，清除登录状态
      window.location.href = '/login'       // 跳回登录页
    }
    return Promise.reject(err)
  }
)
```

效果：所有 API 调用代码都不需要手动加 Token，也不需要手动处理 401，拦截器统一搞定。

### 4.5 API 接口封装 `src/api/auth.ts` 和 `src/api/agents.ts`

把每个后端接口封装成一个函数，并用 TypeScript 声明返回值类型：

```typescript
// 登录：POST /api/v1/auth/login
export const login = (payload: LoginPayload) =>
  client.post<{ code: number; data: TokenResponse }>('/api/v1/auth/login', payload)

// 获取智能体列表：GET /api/v1/agents
export const listAgents = (page = 1, pageSize = 20) =>
  client.get<{ code: number; data: { items: Agent[]; total: number } }>(
    '/api/v1/agents', { params: { page, page_size: pageSize } }
  )

// OpenAI 兼容：GET /v1/models
export const listModels = () =>
  client.get<{ object: 'list'; data: OpenAIModel[] }>('/v1/models')
```

### 4.6 公共布局 `src/components/AppLayout/index.tsx`

整个应用（除登录页外）的骨架：
- **左侧 Sider**：固定宽 220px 的深色导航菜单，包含"智能体管理"和"对话广场"两个入口
- **顶部 Header**：固定置顶，右侧显示当前登录用户名 + 退出登录按钮
- **中间 Content**：通过 `<Outlet />` 渲染当前路由匹配到的子页面

### 4.7 登录页 `src/pages/Login/index.tsx`

包含"登录"和"注册"两个 Tab，核心登录流程：

```
用户点击登录
    ↓
POST /api/v1/auth/login → 获取 access_token
    ↓
GET  /api/v1/auth/me（传入 token）→ 获取用户信息
    ↓
setAuth(token, userInfo) 存入 Zustand store（持久化到 localStorage）
    ↓
navigate('/agents') 跳转到智能体列表
```

> **注意**：`getMe()` 调用时需要把刚收到的 token 显式传入，因为此时 store 还没更新，请求拦截器读不到最新 token。

### 4.8 智能体列表页 `src/pages/Agents/index.tsx`

- 页面加载时调用 `listAgents()` 获取智能体数据
- 用 `Row + Col` 栅格把智能体渲染成响应式卡片（一行 4 个 → 2 个 → 1 个，随屏幕缩放）
- 每张卡片显示：智能体类型（Chat/ReAct/Task）、发布状态、模型名称、版本号
- 底部"开始对话"按钮 → `navigate('/chat/${agent.id}')`

### 4.9 对话页 `src/pages/Chat/index.tsx`（核心页面）

这是功能最复杂的页面，分三个区域：

**顶部信息栏**：显示当前智能体名称、类型、使用的模型，以及"返回"和"清空对话"按钮。

**消息列表区**：
- 用户消息：右对齐，紫色气泡
- AI 消息：左对齐，白色气泡，通过 `react-markdown` 渲染 Markdown（标题、代码块、粗体等）
- 流式输出时，气泡末尾显示闪烁光标 `▋`（CSS 动画实现）
- 工具调用结果：AI 消息下方的灰色展示面板

**底部输入区**：
- 多行文本框，支持自动撑高（最多 6 行）
- Enter 发送，Shift+Enter 换行
- 发送中变为"停止"按钮，点击中断流式输出

## 五、前端如何对接后端（关键技术点）

### 5.1 解决跨域问题：Vite Proxy 代理

**什么是跨域？**
浏览器出于安全考虑，不允许前端页面（`localhost:3000`）直接请求另一个地址（`localhost:8000`）的数据，这就是"跨域"限制。

**解决方案**：Vite 开发服务器充当"中间人"，前端请求发给 Vite，Vite 再转发给后端。

```typescript
// frontend/vite.config.ts
server: {
  port: 3000,
  proxy: {
    '/api': { target: 'http://localhost:8000', changeOrigin: true },
    '/v1':  { target: 'http://localhost:8000', changeOrigin: true },
  },
}
```

效果：
```
前端发出: GET http://localhost:3000/api/v1/agents
Vite 转发: GET http://localhost:8000/api/v1/agents  ← 后端实际收到的
```

前端代码里只需要写 `/api/...`，无需写完整的后端地址。

### 5.2 JWT 认证：Token 自动注入

后端使用 JWT Bearer Token 做认证，每个需要鉴权的接口都要在 HTTP Header 里带上：

```
Authorization: Bearer eyJhbGci...
```

这个工作由 `src/api/client.ts` 的请求拦截器自动完成，所有接口调用都无感知。

### 5.3 流式对话：SSE（Server-Sent Events）

这是最核心的对接难点。普通 HTTP 请求是"一问一答"——等 AI 全部生成完再返回。但我们希望像 ChatGPT 一样逐字输出，需要用 SSE（服务器推流）。

**SSE 数据格式**（后端按 OpenAI 规范输出）：
```
data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","choices":[{"delta":{"content":"你"},"finish_reason":null}]}

data: {"id":"chatcmpl-xxx","object":"chat.completion.chunk","choices":[{"delta":{"content":"好"},"finish_reason":null}]}

data: [DONE]
```

**前端如何读取**：使用原生 `fetch` + `ReadableStream`，一边接收一边解析：

```typescript
const resp = await fetch('/v1/chat/completions', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', Authorization: `Bearer ${token}` },
  body: JSON.stringify({ model: agentId, messages: history, stream: true }),
  signal: ctrl.signal,    // AbortController：用于"停止生成"
})

const reader = resp.body!.getReader()
const decoder = new TextDecoder()
let buffer = ''           // 缓冲区（防止数据包被截断）
let accumulated = ''      // 累积已收到的文字

while (true) {
  const { done, value } = await reader.read()  // 读一块数据
  if (done) break
  buffer += decoder.decode(value, { stream: true })

  const lines = buffer.split('\n')
  buffer = lines.pop() ?? ''   // 最后不完整的行留在缓冲区

  for (const line of lines) {
    if (!line.startsWith('data: ')) continue
    const data = line.slice(6).trim()
    if (data === '[DONE]') break
    const json = JSON.parse(data)
    const chunk = json.choices?.[0]?.delta?.content
    if (chunk) {
      accumulated += chunk
      // 实时更新最后一条消息（触发 React 重渲染 → 界面逐字更新）
      setMessages(prev => {
        const updated = [...prev]
        updated[updated.length - 1] = { role: 'assistant', content: accumulated, streaming: true }
        return updated
      })
    }
  }
}
```

**为什么不用 EventSource API？**
浏览器原生的 `EventSource` 只支持 GET 请求，无法发送请求体（无法传 messages、model 等参数）。我们的接口是 POST，所以必须手动用 `fetch` 解析流。

### 5.4 多轮对话：通过 messages 数组传递历史

OpenAI 协议没有"会话 ID"的概念，历史消息直接放在请求体的 `messages` 数组里：

```typescript
// 每次发消息，都把完整历史一起发给后端
body: JSON.stringify({
  model: agentId,
  messages: [
    { role: 'user',      content: '你好' },
    { role: 'assistant', content: '你好！有什么可以帮你的？' },
    { role: 'user',      content: '现在几点了？' },  // ← 本次新消息
  ],
  stream: true,
})
```

前端维护一个 `messages` 数组，每次发送时整体传给后端，后端不存会话历史，无状态。

### 5.5 停止生成：AbortController

```typescript
const ctrl = new AbortController()
abortRef.current = ctrl   // 存到 ref，供"停止"按钮访问

// 请求时传入 signal
fetch('/v1/chat/completions', { signal: ctrl.signal, ... })

// 用户点击停止
const stopGenerate = () => {
  abortRef.current?.abort()   // 主动中断 HTTP 连接
  setLoading(false)
}
```

`AbortController` 是浏览器原生 API，调用 `abort()` 会立即中断正在进行的 fetch 请求，触发 `AbortError`，代码里捕获该错误并静默处理（不弹错误提示）。

## 六、数据流全景图

```
用户输入消息
    │
    ▼
Chat 组件 sendMessage()
    │  构造 messages 数组（含历史）
    │
    ▼
fetch POST /v1/chat/completions
    │  { model: agentId, messages: [...], stream: true }
    │  Authorization: Bearer <JWT Token>
    │
    ▼  (Vite Proxy 转发到 :8000)
    │
FastAPI openai_router.py
    │  解析 agent_id、调用 AgentRuntime
    │
    ▼
AgentScope Runtime（流式执行）
    │  每生成一个 token 就 yield 一个 chunk
    │
    ▼  SSE 数据流
    │  data: {"choices":[{"delta":{"content":"你"},...}]}
    │  data: {"choices":[{"delta":{"content":"好"},...}]}
    │  data: [DONE]
    │
    ▼
Chat 组件 ReadableStream reader.read()
    │  解析每个 chunk，累积文字
    │
    ▼
setMessages() → React 重渲染
    │
    ▼
用户看到逐字输出的回复
```

### 7.2 打字光标动画

流式输出过程中，在气泡末尾显示闪烁的 `▋`：

```css

  content: '▋';
  animation: blink 0.8s step-end infinite;
}
@keyframes blink {
  50% { opacity: 0; }
}
```

组件里通过条件加类名触发：
```tsx
<div className={`msg-bubble assistant${msg.streaming ? ' typing-cursor' : ''}`}>
```

### 7.3 Markdown 渲染

AI 经常返回带格式的 Markdown，如：
````

```python
````

用 `react-markdown` 一行代码完成渲染：
```tsx

```

配合 `index.css` 中对 `.msg-bubble code` 和 `.msg-bubble pre` 的样式定义，代码块会有灰色背景和等宽字体。

## 九、总结：前端做了哪些适配工作

1. **协议适配**：后端暴露的是 OpenAI 兼容格式（`/v1/chat/completions`），前端直接按 OpenAI 格式构造请求，`model` 字段传 `agent_id`，无需额外适配层。

2. **跨域处理**：通过 Vite proxy 把 `/api` 和 `/v1` 请求透明转发到后端，开发阶段零配置跨域。

3. **认证处理**：Axios 拦截器自动注入 JWT Token，401 自动跳回登录页，业务代码无感知。

4. **流式消费**：手动用 `fetch + ReadableStream` 解析 SSE，按 OpenAI delta 格式逐 chunk 累积文字，实时更新界面，模拟 ChatGPT 打字效果。

5. **登录时序修复**：登录完成时先用刚拿到的 token 显式请求用户信息，再写入 store，避免了 `getMe()` 因 store 未更新而 401 的时序 bug。

6. **多轮对话**：前端自己维护 `messages` 历史数组，每次请求时整体发送，与后端无状态设计保持一致。
